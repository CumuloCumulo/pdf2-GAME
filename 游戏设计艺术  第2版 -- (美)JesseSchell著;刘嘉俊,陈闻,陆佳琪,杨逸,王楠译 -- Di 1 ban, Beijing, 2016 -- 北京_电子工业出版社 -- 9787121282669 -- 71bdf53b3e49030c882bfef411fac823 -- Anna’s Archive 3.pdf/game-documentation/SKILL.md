# 游戏文档系统

## 何时使用

- 启动新的游戏开发项目需要建立文档体系
- 需要记录设计决策和理由
- 与团队成员沟通设计理念
- 追踪项目资产和进度
- 为不同受众创建合适的文档

## 核心原则

### 没有神奇的模板！

Jason VandenBerghe的洞察："游戏设计文档的问题在于，你写下的那一刻，它们就过时了。设计文档表达你目前认为会对游戏有帮助的东西......直到你看到你的想法付诸行动，你不知道它们是否会奏效。"

文档不是规格说明、脚本或蓝图——它们是想法。

## 文档的两大目的

### 1. 备忘
- 人类记忆很差
- 游戏有数千个关于如何和为什么工作的决定
- 在2周内200个设计决定后容易忘记即使是绝妙的解决方案
- 记录决定防止重复解决相同问题

### 2. 沟通
- 与团队成员分享设计想法
- 创建双向对话（不是单向的）
- 一旦写下，其他人可以反对或改进
- 更快聚集更多头脑来发现和修复设计缺陷
- Google Docs因此成为标准

## 如何开始

1. 从简单的粗略要点列表开始，列出你想要在游戏中的东西
2. 随着列表增长，设计问题会出现——把它们写下来
3. "完善你的设计" = 回答这些问题
4. 记录每个决定和理由
5. 想法、计划、问题、答案自然地落入类别
6. 文档围绕你独特的游戏有机地增长
