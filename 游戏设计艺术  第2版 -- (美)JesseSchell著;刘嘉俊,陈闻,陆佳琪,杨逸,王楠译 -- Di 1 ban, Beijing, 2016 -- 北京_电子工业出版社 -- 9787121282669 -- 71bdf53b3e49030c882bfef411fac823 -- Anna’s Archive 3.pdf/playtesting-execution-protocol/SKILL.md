# 游戏测试执行协议

## 适用场景
- 游戏开发团队进行可玩性测试
- 设计团队验证游戏机制
- 测试引导者执行用户测试

## 前提条件
- 测试目标已定义
- 测试人员已招募
- 测试地点已选定

## 组件一：开发者出席决策

1. **出席风险**：开发者在情感上投入游戏，可能无意识地引导测试人员避开问题区域。
2. **风险评估**：这种可能性非常高。
3. **建议**：如果无法控制自己并保持客观冷静，请勿出席，以确保测试人员在“自然纯粹”状态下游戏。
4. **反驳论点**：直接观察能提供问卷和录像无法获得的信息。
5. **最终建议**：尽可能出席，但积极防止向测试人员泄露信息。

## 组件二：测试前开场白

1. **何时保持沉默**：如果测试玩家是否能独立理解游戏信息，让游戏自我介绍。
2. **大多数测试**：提供足够信息让玩家开始游戏。
3. **关键警告**：几句不慎的言语可能破坏玩家的未知状态。
   - 示例：告诉玩家目标是击败邪恶的克洛诺斯可能导致他们从一开始就寻找最终目标，错过本应自然探索的重要细节。
4. **最佳实践**：
   - 记录测试开始前对测试人员说的所有内容。
   - 提前写下开场白，确保不同批次测试人员获得一致信息。
   - 如果发生意外情况，与记录的开场白对比以查找原因。
5. **迭代优化**：
   - 多次测试后，逐步调整信息和指令。
   - 这里删一个词，那里加一个短语。
   - 润色开场白直到非常清晰和精确。
   - 记录所有更改——这成为游戏教程的基础。
   - 通过此过程精心制作的教程是“业界良心”。

## 组件三：测试期间观察焦点

1. **常见方法**：跟随玩家目光，盯着屏幕（特别是主机游戏）。
   - 理由：从玩家第一人称视角观察。
2. **作者推荐方法**：关注玩家的面部表情。
   - 偶尔瞥视屏幕以了解正在发生的事情。
   - 主要注意力集中在玩家脸上。
   - 理由：玩家的感受比他们的行为更重要。
   - 面部表情揭示测试后问答或调查无法获得的游戏体验信息。
3. **技术来源**：街头艺人经验。
   - 只有在表演结束后传帽子才能赚钱。
   - 必须让观众保持参与直到最后才能吃顿像样的晚餐。
   - 学会解读人群情绪变化并相应调整表演。
   - 延长受欢迎的段落，跳过无聊的部分。
4. **多摄像机设置（现代解决方案）**：
   - 在不同角度放置多个摄像机。
   - 将素材拼接在一起。
   - 捕捉：游戏画面、面部表情、手部动作。
   - 测试结束后全面回顾。

## 输出格式
执行游戏测试，包含受控的开发者出席、一致的开场白和以面部表情为核心的观察方法。
