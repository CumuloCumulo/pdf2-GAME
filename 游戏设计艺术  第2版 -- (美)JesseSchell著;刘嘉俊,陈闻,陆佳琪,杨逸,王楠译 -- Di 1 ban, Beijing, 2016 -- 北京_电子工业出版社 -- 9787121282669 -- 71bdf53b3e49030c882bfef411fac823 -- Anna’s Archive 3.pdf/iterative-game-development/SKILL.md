# 迭代式游戏开发

## 何时使用

- 启动新的游戏或软件开发项目
- 需要在瀑布模型和迭代模型之间做选择
- 项目存在技术或设计上的不确定性风险
- 需要通过原型验证设计假设
- 团队需要建立敏捷开发流程

## 核心方法论

### 迭代法则

迭代法则回答两个核心问题：
1. **如何让每次迭代有意义？** → 评估并消除风险
2. **如何尽可能快地迭代？** → 构建许多粗糙的原型

### 螺旋模型核心步骤

1. 想出一个基础的设计
2. 找出设计中最大的风险
3. 建立原型消除这些风险
4. 测试这个原型
5. 基于原型结论作更详细的设计
6. 回到第2步，重复直到系统完成

### 敏捷开发核心要素

1. **灵活的目标**：计划改变计划，而非抵抗变化
2. **优先级列表**：按重要性排序功能，不保证全部完成
3. **冲刺（Sprints）**：数周为一个周期，每个冲刺交付可用成果
4. **每日站会**：10-15分钟，每人说明昨天完成什么、今天计划什么、遇到什么问题
5. **演示日**：冲刺结束时团队面对面测试结果
6. **回顾会议**：讨论什么做对了、什么做错了、下次如何调整

## 风险评估与消除流程

### 第一步：风险识别
召集团队进行全面风险分析，列出所有可能危害项目的风险。

### 第二步：风险优先级排序
- 按严重程度和可能性排序
- 考虑风险之间的依赖关系
- "上游"风险（可能使其他工作无效的风险）获得最高优先级

### 第三步：通过原型消除风险
在投入主要资源前，构建小型原型来减少或消除每个风险。

### 16号透镜：风险消除

**关键问题：**
- 是什么让这个游戏变得平庸？
- 我们怎样防止这样的风险发生？

**警告：** 风险管理很难，必须面对不想遇到的问题，抵制只关注有信心部分的诱惑。

## 原型技术

### 技术一：回答问题
每个原型都应该回答一个或多个明确的问题。如果无法描述问题，原型可能在浪费时间。

### 技术二：忘记质量
- 唯一关注点：原型是否解决了问题？
- 更快的解决方案 = 更好的原型
- 警告：打磨原型可能有害，粗糙的原型更容易暴露问题

### 技术三：不要太过依恋
- 第一版系统不是成品，是要被丢弃的原型
- 以"一切都是临时的"心态开始原型开发
- 每个原型都是学习机会

### 技术四：区分原型优先级
解决最大风险优先。如果一个原型的结果可能使其他原型失去意义，这个"上游"原型优先级最高。

### 技术五：有效的并行原型
系统工程师构建技术原型、美术构建美术原型、设计师构建玩法原型——同时进行。

### 技术六：并不总需要数字化
- 桌面纸原型可以快速构建并达到相同的玩法验证效果
- 回合制游戏易于转换，实时游戏可转为回合制或使用节拍器模拟

### 技术七：无需交互
简单的草图和动画也可以帮助回答玩法问题。

### 技术八：选择快速迭代引擎
选择可以在运行时修改代码的引擎，实现"粘土建模式"开发而非"烤面包式"开发。
