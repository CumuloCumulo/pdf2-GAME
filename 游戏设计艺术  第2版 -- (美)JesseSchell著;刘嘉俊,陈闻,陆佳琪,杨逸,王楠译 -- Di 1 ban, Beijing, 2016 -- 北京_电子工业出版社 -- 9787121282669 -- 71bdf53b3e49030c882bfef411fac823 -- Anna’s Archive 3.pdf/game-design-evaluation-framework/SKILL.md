# 游戏设计评估框架

## 总体规则

所有八项测试都必须通过，设计才能被认为是"优秀的"。如果任何测试失败，修改设计并**重新运行所有测试**（修改可能通过一项测试但破坏其他测试）。

## 八项测试

### 测试 1：艺术冲动

- 最个人的测试
- 设计师问："这个游戏对我来说感觉对吗？"
- 如果是 → 通过；如果否 → 需要更改
- 设计师和团队的直觉都很重要
- 直觉并不总是正确的，但其他测试可以平衡这一点
- **关键问题**："这个游戏看起来不错吗？"

### 测试 2：人群特征

- 游戏有特定的目标受众
- 受众分类：年龄组、性别或其他类别（如高尔夫爱好者）
- 考虑设计是否适合目标受众
- **关键问题**："我们的目标受众喜欢这个游戏吗？"

### 测试 3：体验设计

- 通过以下方式创造出色的体验：美学、兴趣曲线、共鸣主题、游戏平衡
- 书中的许多透镜与体验设计相关
- 游戏必须能够通过这些透镜的审查
- **关键问题**："这个游戏设计得好吗？"

### 测试 4：革新

- 新游戏必须包含玩家以前没见过的东西
- 游戏是否独特是主观的，但至关重要
- **关键问题**："这个游戏是否与众不同？"

### 测试 5：商业和市场

- 游戏商业**就是**商业 - 设计师必须纳入这一现实
- 关键考虑因素：
  - 主题/故事是否吸引玩家？
  - 是否容易理解 - 玩家能否从概览中掌握游戏？
  - 消费者对此类型有什么期望？
  - 此游戏与市场上的类似游戏相比如何？
  - 开发成本是否太高而无法盈利？
- 讽刺的是：最初的设计想法在考虑此测试时可能被证明完全站不住脚
- **关键问题**："这个游戏能够盈利吗？"

### 测试 6：工程

- 游戏想法只是一个想法，直到工程完成
- 想法并不总是与约束条件（可能/可行）一致
- 必须回答："我们如何构建这个游戏？"
- 技术限制可能阻止原始愿景
- 新手设计师经常对工程约束感到沮丧
- 工程测试可以将游戏引向新方向 - 为实现而创建的功能可能不是源自设计师
- 在此测试期间产生的想法很有价值，因为它们被确认为可行
- **关键问题**："这个游戏在技术上是否具备可行性？"

### 测试 7：社交/社区

- 一个有趣的游戏可能还不够
- 一些设计目标需要：强大的社交元素、病毒式传播、社区形成
- 游戏设计强烈影响这些因素
- **关键问题**："这个游戏完成了我们的社交或社区目标吗？"

### 测试 8：玩法测试

- **最重要的测试**
- 想象游戏 ≠ 实际玩游戏 ≠ 目标受众玩游戏
- 尽快让游戏达到可玩状态
- 只有看到实际游戏玩法时，你才知道需要什么重大更改
- 当你更好地理解机制和玩家心理学时，测试目标可能会改变
- **关键问题**："游戏测试者是否享受这个游戏？"

## 附加测试

- 一些设计需要额外的测试（例如，教育游戏必须回答："这个游戏实现了其教育目标吗？"）
- 不要省略必要的额外测试

## 测试调整规则

- 在设计过程中，可能需要调整测试
- 示例：最初针对 18-35 岁男性，但最终设计更适合 50 岁以上女性
- 当设计受到约束时，更改测试是可以接受的
- 目标：找到一种通过所有八项测试的方法，无论是通过更改测试还是设计

## 何时使用

- 当评估游戏设计提案时
- 当验证游戏原型时
- 当审查设计文档时
- 当需要确定设计是否准备好进入下一阶段时
